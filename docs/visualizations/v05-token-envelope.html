Caption (for the doc)

Token envelope encryption per ADR-0002. Each seal generates a fresh 24-byte nonce; the AEAD construction (XChaCha20-Poly1305) produces an authenticated nonce ‖ ciphertext ‖ tag bundle. The 32-byte master key lives in TOKEN_MASTER_KEY only — never persisted in DB. Open returns either plaintext (tag verified) or a tamper-detection error; partial-decrypt-on-bad-tag is impossible by AEAD construction. Master-key rotation requires a re-encrypt drill (Incident C in runbook); PCO-57 tracks the long-term envelope-with-per-row-data-keys refactor.

Design notes

The most important asymmetry — master key always-in-env vs data always-in-row — is shown by the red trust island floating above both columns and feeding both flows with dashed red arrows. The envelope structure is laid out as three concrete byte-segments (nonce / ciphertext / tag) so "envelope" is literal, not metaphorical.
