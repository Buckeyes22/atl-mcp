# Domain Vocabulary

> **TL;DR:** The deeper glossary. Defines state-machine states, every domain type, design patterns used, and key terms that appear in the codebase. Companion to [`README.md`](README.md) (which covers cross-cutting vocabulary). When a term feels under-defined in code: check here.

The `README.md` glossary covers vocabulary an outsider needs. This doc covers vocabulary an engineer needs to read the code fluently.

## State-machine states (`ProjectState`)

13 states from v6 §6:

### `DRAFT_INTAKE`
Initial state. A project ID exists but no intake has been captured. Transitions to `INTAKE_RECEIVED` on `project_intake_create`.

### `INTAKE_RECEIVED`
Intake captured (raw markdown / UIO doc). Transitions to `BLUEPRINT_DRAFTED` on `project_blueprint_generate` start.

### `BLUEPRINT_DRAFTED`
Blueprint generated by sampling, not yet validated. Transitions to `BLUEPRINT_VALIDATED` (success) or `VALIDATION_FAILED` (adversarial triplet rejected).

### `BLUEPRINT_VALIDATED`
Blueprint passes adversarial verification triplet. Persisted; ready for planning. Transitions to `PROVISIONING_PLANNED` on `project_provision_preview`.

### `PROVISIONING_PLANNED`
Planner has produced an `ArtifactPlan`. Transitions to `PROVISIONING_PREVIEWED` on operator-visible preview.

### `PROVISIONING_PREVIEWED`
Operator has reviewed the plan but not approved. Transitions to `PROVISIONING_EXECUTED` on approval.

### `PROVISIONING_EXECUTED`
Plan executed successfully (or partially — re-runs are idempotent). Transitions to `CONTEXT_PACKED` when context packs are generated.

### `CONTEXT_PACKED`
Context packs available for build agents. Transitions to `READINESS_CHECKED` on `readiness_validate`.

### `READINESS_CHECKED`
Readiness rubric run; verdict known. Transitions to `READY_FOR_BUILD` if verdict is "Ready"; stays here otherwise (with warnings).

### `READY_FOR_BUILD`
Project is in a state where build agents can begin. Transitions to `BUILD_IN_PROGRESS` on first build-agent activity.

### `BUILD_IN_PROGRESS`
Build agent is doing work. Webhook ingestion updates project graph. Transitions to `DRIFT_DETECTED` if external state diverges from blueprint, or `ARCHIVED` on completion.

### `DRIFT_DETECTED` (side state)
External state has diverged from blueprint (e.g., issues moved that the blueprint didn't anticipate). Cycles back to `READY_FOR_BUILD` after re-sync.

### `VALIDATION_FAILED` (terminal-ish)
Blueprint failed adversarial verification. Operator can edit intake and re-attempt; but this state itself is terminal for that blueprint draft.

### `ARCHIVED` (terminal)
Project is closed; no further automation. Audit chain entries persist; data retained per [`../05-data/retention.md`](../05-data/retention.md).

## Domain type catalog

### `ProjectBlueprint`

Root aggregate. Composed of:
- `ProjectIntake` (the raw source).
- `Requirement[]` (atomic items).
- `Feature[]` (high-level features).
- `EpicPlan[]` (release-level epics).
- `BlueprintExtras` (sub-aggregate: architecture, security, testing, release).
- Metadata (created/updated timestamps, version, classification).

### `ProjectIntake`

The captured source content. Has a `source.kind` discriminator: `raw_markdown` | `uio_document` | `uio_file_upload`.

Always has `sourcePins[]` — pointers back to the original content for traceability (chunks, line offsets, document IDs).

### `Requirement`

Atomic unit of project intent. Has:
- `title` — short imperative.
- `description` — full prose.
- `acceptanceCriteria[]` — testable assertions.
- `tags[]` — for grouping.
- `priority` — must / should / could.
- `sourcePins[]` — traceability.

### `Feature`

Mid-level grouping: a coherent capability composed of multiple requirements. Used in blueprint structuring.

### `EpicPlan`

Release-time grouping: a milestone-level slice with stories, timeline (start, target), and goals.

### `BlueprintExtras`

Sub-aggregate covering parts of the blueprint that aren't requirements/features/epics:
- Architecture summary.
- Security / privacy notes.
- Testing strategy.
- Release plan.

### `SourcePin`

Pointer to upstream source. Has:
- `kind` — markdown_block / uio_chunk / file_offset.
- Identifier (chunk ID, line range, etc.).
- Confidence (how strong is the link).

Source pins enable trace-back: from a requirement to the prose that introduced it.

### `ProjectProfile`

Output of preflight. Per-target capability snapshot:
- Jira: project type, issue types, custom fields, workflows.
- Confluence: storage formats supported, macro inventory, templates available.
- VCS: workspace, repo, default branch, permissions.
- Plus warnings array.

TTL-scoped; refresh on permission webhook or operator-initiated.

### `AclEntry`

ACL decision cache. `(projectId, artifactRef, principalId)` → `(decision, source, expiresAt)`.

### `AuditEntry`

Signed audit log entry. See [`../05-data/audit-trail.md`](../05-data/audit-trail.md).

### `PolicyDecision`

`(effect, obligations, confidence, reasons)`. Output of `policyDecisionLayer.evaluate()`. Persisted in `policyDecisions` (queryable) AND in `auditEntries` (tamper-evident).

### `ContextPack`

Bounded redacted context for a build agent. Has:
- `regenerationKey` — deterministic hash of inputs.
- `targetModel` — which model context size we're targeting.
- `tokenCount` — actual token count.
- `content` — the pack body (multi-section: blueprint summary, issue context, related artifacts, etc.).

### `TenantScope`

Single-tenant runaway / multi-tenant seam. Has `tenantId` + optional `instanceUrl`. v1: always `defaultTenantScope()`.

### `TokenBudget`

Per-request and per-session limits. Has `perRequest`, `perSession`, `overflow` ("reject" | "truncate").

### `MCPSessionProfile`

Per-MCP-session metadata: `sessionId`, declared `capabilities`, `tokenBudget`, `targetModel`, host features (sampling, thinking).

### `Confidence`

Dual representation: `numeric` (0.0-1.0) + `categorical` ("low" | "medium" | "high"). Used in PolicyDecision and elsewhere where uncertainty matters.

### `ArtifactRef`

Uniform artifact identifier. `kind` ("jira-issue" | "confluence-page" | "vcs-branch" | "vcs-pr"), `key`, `displayName`, `url`. Used wherever artifacts cross system boundaries.

### `TraceLink`

Source-pin-style traceability for blueprint outputs. `(tenantId, projectId, sourceRef, targetRef)` plus metadata.

### `ProjectGraph`

(M10+) Aggregated change-graph view of the project's external state. Used by webhook ingestion to detect drift.

## Design patterns

### Repository pattern

Every storage interaction goes through a repository (`projectRepository`, `auditEntriesRepository`, etc.). Repositories enforce tenant scope; raw SQL outside them is forbidden.

### Adapter pattern (policy decision)

The `policyDecisionLayer` holds the contract; adapters implement specific rules. v1 has `codePolicyAdapter`; future could have OPA, Cedar, etc. Adapters all return `PolicyDecision` shape.

### Test double (token encryption)

`tokenEncryption.testDouble.ts` provides a fake that preserves the seal/open contract without doing crypto. Lets non-encryption tests skip the crypto path.

### Adversarial verification triplet

Three-pass validation: emit, critique-by-different-prompt, accept-or-reject. Used on blueprint outputs (M4+). v6 §18.1.

### Tool-collapse pattern

One MCP tool with action enum > many narrow tools. Smaller surface for build agents. v6 §14.

### Single-message Task dispatch

When spawning sub-agents in Claude Code, do it in a single assistant message. Multi-message dispatch is sequential. CLAUDE.md, v6 §24.6.

### Iron-law pattern

Two non-negotiable rules: verify-before-claim-done; test-first for new behavior. F-106 (superpowers).

### Hash-chain + signature pattern

For audit log integrity: each entry hashes to the next; each entry is signed; key registry is versioned in git. ADR-0005.

### Envelope encryption (planned, PCO-57)

Per-row data keys, master key encrypts data keys. Reduces blast radius of master-key rotation. v1 has single-master-key; envelope is post-v1 work.

### Capability negotiation

MCP session-start handshake to determine the intersection of supported capabilities. v6 §2.2.

### Tenant scope assertion

`assertTenantMatches(scope, attemptedAccess)` at every repository read/write. Throws on mismatch. Single guarantee that single-tenant is preserved.

## Common code idioms

### `factory` functions

Factories take dependencies, return configured instances:

```typescript
const myRepo = projectRepository(db, tenantScope)
```

Avoids classes-with-DI-containers; functional style.

### Discriminated unions

For state machines + variants:

```typescript
type ProjectState =
  | { kind: "DRAFT_INTAKE" }
  | { kind: "INTAKE_RECEIVED"; intakeId: string }
  // ...
```

Compiler enforces exhaustive switch.

### Branded types

For type-safe identifiers:

```typescript
type ProjectId = string & { readonly __brand: "ProjectId" }
```

Prevents passing a Bitbucket repo ID where a Project ID is expected.

### `assertTenantMatches`

Pattern at every repository boundary. Throws `TenantScopeViolationError` on mismatch.
