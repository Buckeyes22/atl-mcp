# atl-mcp conformance baseline — 2026-04-26

> **What this is.** First scoring of atl-mcp against the conformance rubric in [`docs/conformance/rubric.md`](rubric.md). Reference point for "is the work getting better or worse?" across future M11 iterations.

## Tests in scope

The 33 admin integration tests that exercise the loopback `/mcp` transport against in-memory PGlite + the typed admin tool surface:

## Per-category scores

### `mcp_protocol_compliance` — **5/5**

The transport tests in `transportAndUi.test.ts` exercise the full MCP lifecycle: `initialize` returns a session id; `notifications/initialized` is accepted; `tools/list` returns the registered tool catalog; `tools/call` round-trips with the right `content` + `structuredContent` shape. Static asset host (`/ui/`) is verified at the same time. Path traversal is blocked. Edge cases like missing `Accept` header would surface in production but aren't tested here — minor gap, doesn't move the score.

### `tool_correctness` — **4/5**

Read tools (13) all assert the structured shape against fresh DB and against seeded blueprints (`lifecycleTools.test.ts`). Write tools (6 + 5 lifecycle execute calls) assert the right repository mutation, follow-up policy decisions, and sampling round-trip. The one weakness: live-provider behavior (Jira issue creation, Confluence page creation, real Bitbucket repo creation) is mocked or `dataLimited`'d in the test fixture. The **Bitbucket REST tests** (`bitbucketRestProvider.test.ts`, 9 tests) cover the wire shape against a recorded fetch mock; that's good but not equivalent to a live sandbox. -1 for that.

### `audit_evidence` — **5/5**

Every write tool in the admin surface calls `appendOperatorAudit` and returns the `auditEntryId` in its result. The `writeTools.test.ts` suite asserts these IDs are real UUIDs and the chain accumulates correctly. The audit chain integrity verifier is wired (`admin.audit.verify`) and runs in the read-tools test against a fresh chain. The signing key id is exposed via `admin.audit.head` and asserted in the handoff bundle test. ed25519 + JCS canonicalization is exercised by the dedicated `auditChain` tests.

### `policy_gating` — **3/5**

`writeTools.test.ts` confirms `admin.policy.approve` writes a follow-up decision and that the original `require_approval` decision was the parent. The lethal-trifecta detector and confidence-threshold gates are stubbed in the policy adapter — tests confirm the structure but not that production decisions actually flow through them in the M5 planner. The agent-facing `provisioningWorkflow` does call `policy.decide()` before each Jira preview action, but the admin-side fast-path (`jiraIssueTreeWorkflow`) does NOT — that's a real gap that would need M5 hardening before this score moves up. -2 for the missing admin-side gate.

### `data_limited_honesty` — **5/5**

Every Phase 4 data-limited tool (`admin.alerts.*`, `admin.slos.list`, `admin.capacity.get`, `admin.dr.*`, `admin.config.flags.toggle`, `admin.secrets.rotate.*`) returns a structured `dataLimited.reason` instead of fake values or thrown errors — directly tested by the read/write/lifecycle tests. The lifecycle execute tools (`jira/confluence/vcs.execute`) return `dataLimited` honestly when the corresponding provider isn't configured — verified in `lifecycleTools.test.ts`. The UI surfaces these via `<DataLimited reason />` and `<DataLimitedBanner reason />`.

### `idempotency_and_failure` — **3/5**

`vcsRepoScaffoldWorkflow.test.ts` exercises the round-trip; `createRepository` returns `alreadyExisted: true` when the slug exists (per the bitbucket fixture test). Jira `createIssue` uses an idempotency key tied to `blueprint.id + node.id + blueprint.blueprintVersion` — re-running with the same version is idempotent at the schema level, but provider-side idempotency depends on Bitbucket / Atlassian honoring the key. Confluence `createPage` uses an idempotency key per `space:templateSlug:title` — same caveat. Failure-path tests are thin: there's no explicit test that asserts a 500 from the provider produces a structured error result with an audit entry rather than throwing. -2 for the failure-path coverage gap.

## Top 3 highest-value tests to add

1. **Admin-side policy gate.** Add a test asserting that `admin.lifecycle.jira.execute` consults the policy adapter for each story before creating it (or document explicitly that the admin path bypasses the gate by design and route operator action through a separate review tool). Lifts `policy_gating` from 3 → 5.
2. **Provider failure path with audit entry.** Mock a 500 response from Bitbucket's `POST /repositories`, run `admin.lifecycle.vcs.execute`, assert the result has `ok: false` + `dataLimited.reason` containing the upstream message + a real `auditEntryId` recording the failure. Same shape for Jira (5xx on `createIssue`) and Confluence (401 on `createPage`). Lifts `idempotency_and_failure` from 3 → 5.
3. **Live sandbox round-trip.** A nightly conformance test that hits a real Bitbucket sandbox + Jira sandbox + Confluence sandbox with a synthetic blueprint, asserts the issues / pages / repo land, then cleans up. Lifts `tool_correctness` from 4 → 5 and gives `idempotency_and_failure` a real-world reference point.

Adding all three would put the aggregate at 29-30 — solidly in "ship" territory.
