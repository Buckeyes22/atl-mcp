# atl-mcp — 5-Minute Tour

> **For:** technical recruiter handoff, staff engineer pre-screen, end of a coding interview where you have a few minutes to show something else.
> **Live links:** [PCO Jira project](https://lateapexllc.atlassian.net/jira/software/projects/PCO/boards/1) · [ACO Confluence space](https://lateapexllc.atlassian.net/wiki/spaces/ACO).

---

## (0:00–0:30) Pitch and frame

> "I built an MCP server that turns raw project requirements into agent-ready Jira, Confluence, and Bitbucket workspaces. The Jira project and Confluence space you're about to see were generated by it, from its own requirements."

That's the entire pitch. The dogfooding frame is the demonstration.

## (0:30–1:30) Architecture

Open [`architecture.md`](architecture.md) (or the Confluence Architecture Overview page). Read the TL;DR. Show the system context diagram.

Name the three trust boundaries:

1. External callers (build agents authenticate at MCP session start; operators authenticate via admin REST).
2. Server (writes gate through the policy decision layer; every state change appends an audit entry).
3. External systems (Atlassian Cloud and Bitbucket Cloud; tokens encrypted at rest, plaintext only in memory).

Mention the dual-port HTTP transport (admin REST + MCP streamable HTTP, v6 §22). Mention the policy decision layer ([`src/security/policyDecisionLayer.ts`](../../src/security/policyDecisionLayer.ts)) and the audit chain ([`src/storage/schema/auditEntries.ts`](../../src/storage/schema/auditEntries.ts), v6 §30.1, ADR-0005). Don't drill in unless asked.

## (1:30–3:00) Generated Jira project

Switch to [the PCO board](https://lateapexllc.atlassian.net/jira/software/projects/PCO/boards/1).

Note: 8 build epics + 5 flagship stories + ~50 filler tickets + 13 subtasks; plus 6 future-work epics + 19 tasks + 63 subtasks tracking unfinished features (added 2026-04-27) distributed across components. Components mirror `src/` directories.

Open the **Audit + Policy Enforcement** epic (PCO-7) — it's the most architecturally interesting because it ties two concerns (policy + audit) together.

Open the flagship story under it — **PCO-9: Implement audit chain hash linkage with ed25519 signature**. Read the description. Note the linked ADR-0005 and v6 §30.1 references. Note the five Given/When/Then acceptance criteria including the genesis-block edge case and the key-rotation case.

**The dogfooding line:** _"All of this — the structure of the project, the issue types, the workflow, the components, even this ticket's shape — came out of running the orchestrator on its own profile. None of this is hand-curated."_

## (3:00–4:00) Confluence space

Switch to [the ACO space](https://lateapexllc.atlassian.net/wiki/spaces/ACO).

Show the page tree on the sidebar — 35 pages organized into 8 sections.

Open the **Operational Runbook** page. Show the "Common Incidents" section with three documented incidents (M0 stdout leak, migration shape mismatch, encryption key rotation).

Open the **Audit Findings + Remediation Summary** page. Don't read individual findings; just show the table (15 findings). Then say: "this is the page I think is most worth your time. Let's go there."

## (4:00–5:00) Self-critique

Stay on the Audit Findings page. Pick **F-01** (the M0 stdout leak). Read aloud:

> "I shipped M0 with a `console.log` in `src/mcp/sessionCapabilities.ts` and didn't notice until M1 added the `lint:no-stdout` check. The first MCP client connection failed silently because stdout corruption is invisible to the writer."

Then the lesson:

> "This is a clean example of why protocol-level invariants must be encoded as automated checks before any production code that depends on them. The lesson generalized into the project's iron law: invariants are tooling, not vibes."

End there.
