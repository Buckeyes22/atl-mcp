# How this project was built

> **Read time:** ~5 minutes. Verbal version: ~90 seconds.
> **Why this page exists:** senior engineering interviewers in 2026 have a reasonable question — how much of what you're looking at did I write, and how much was AI-assisted? This page answers it directly.

---

## The 90-second answer

I designed the system architecture and wrote the v6 spec. I orchestrated the build through Claude Code following the build prompts in v6 §29 — every milestone has its own prompt. The Jira project (PCO) and Confluence space (ACO) are the system's *output*, not authored content. Design and judgment are mine; production code is heavily AI-assisted following test-first discipline; generated content is system output.

If you want depth, read on. The categories are: **Designed**, **Orchestrated**, **Generated**.

---

## Designed (by me, with no AI authorship)

These are the decisions and artifacts where I held the pen, weighed the tradeoffs, and recorded the call.

- **The choice to build atl-mcp at all**, and what its scope is. The original problem framing in [`docs/build-orchestration.md`](../build-orchestration.md) and the dogfooding strategy as the demo frame.
- **The v6 spec evolution from v5.** See v6 §0 review summary for the explicit diff: what changed, what was reconsidered, what got dropped. The discipline of writing a v6 review summary instead of overwriting v5 is itself a judgment call.
- **The choice of Atlassian + Bitbucket Cloud as the v1 surface**, and the explicit deferral of GitHub, GitLab, Linear, and Bitbucket Data Center / Server (v6 §3 non-goals).
- **The selection of 10 ADRs** in [`docs/adr/`](../adr/) — what to capture as a decision record vs. what to leave inline. Specifically:
  - ADR-0001 (pglite for dev) — chosen because real Postgres semantics matter and sqlite would have caused divergence.
  - ADR-0002 (token encryption with @noble/ciphers) — chosen because audited primitives + no native build deps.
  - ADR-0003 (Confluence storage default ADF, flagged) — accepted the dual-renderer cost because storage-format-only loses too much.
  - ADR-0004 (Bitbucket app password vs OAuth) — chose simpler ops over OAuth's refresh dance.
  - ADR-0005 (audit signing pipeline) — chose hash-chain + ed25519 + git-ref registry over append-only-with-snapshots.
  - ADR-0006 (operator control plane admin MCP tools) — chose loopback-only admin tools + edge auth gate over baking auth into the application layer.
  - ADR-0007 (project-scoped persistent agent memory) — chose project-scoped retain/recall/reflect/forget with vector index over per-session-only memory or global cross-project memory.
  - ADR-0008 (frontend plus MCP shared workflow) — chose JSON-RPC-over-loopback for the operator UI's admin tools rather than a parallel REST surface.
  - ADR-0009 (GitHub v1 after Bitbucket parity) — chose to land Bitbucket fully before adding GitHub rather than partial parity across both providers.
- **The readiness rubric in v6 §17**, including the 6-category deterministic score, the 4-tier LLM-judged verdict, the weighted project- and issue-level rubrics, and the 5-section skill format. This section is the densest piece of original design in the spec; it answers "what does ready mean?" with measurable criteria.
- **The failure-mode taxonomy in v6 §30.4** and the fix-type taxonomy in §30.5. These are reference data that build agents consume; getting them right was a research-and-synthesis exercise.
- **The audit chain design in v6 §30.1 + ADR-0005**, including the genesis-block treatment (chicken-and-egg case, documented as a special verifier case) and the git-ref-versioned key registry choice.
- **The portfolio strategy.** The decision to use atl-mcp to seed atl-mcp's own project is the most leveraged decision in the whole portfolio. It compresses the entire pitch into one sentence. That choice is mine.
- **The 8-epic project structure** documented in [`interviewer-walkthrough.md`](interviewer-walkthrough.md). 6–8 epics is a sweet-spot judgment call; the specific 8 are mapped to v6's surface areas plus a meta-epic for the portfolio itself.
- **The CLAUDE.md operating rules.** No stdout from `src/`, the iron laws (no done-claim without evidence, no production code without failing test first), single-message Task dispatch. Each rule has a reason and a proof.

## Orchestrated (by me, executed via Claude Code following test-first discipline)

These are artifacts where I made the structural calls and wrote the prompts, but the production code was generated by Claude Code following those prompts.

- **Implementation of milestones M0 through MN**, each following its build prompt in v6 §29. The prompts are not generic — each names the deliverables, the iron law constraints, the test-first requirement, and the file paths to touch.
- **The iron law in CLAUDE.md operating rules.** No production code without a failing test first. This was followed in every milestone. A skeptical reader can verify by checking the git history — tests land in the same commit (or earlier) as the production code under test, and the tests fail at the moment the production code is absent.
- **The structural discipline of CLAUDE.md operating rules:**
  - No stdout from `src/` (enforced by [`scripts/lint-no-stdout.mjs`](../../scripts/lint-no-stdout.mjs) and [`tests/lint/no-stdout.test.ts`](../../tests/lint/no-stdout.test.ts)).
  - Single-message Task dispatch for parallel sub-agents.
  - No inventing skills, hooks, or plugins outside the v6 spec.
- **Code review.** Claude Code reviews code it has written. I review it back. Disagreements get resolved by checking against the v6 spec — the spec is the tiebreaker, not either of our opinions.
- **ADR drafting.** I make the decisions; Claude Code drafts the MADR-format prose; I edit. This blend is honest because the decision (the load-bearing part) is mine.

A skeptical reader can verify the orchestration boundary by reading any single source file alongside the corresponding build prompt in v6 §29. The shape of the code matches the prompt's description. That alignment is intentional — predictable orchestration is the goal.

## Generated (system output, not authored content)

These are artifacts that the orchestrator (or, where the executor isn't yet implemented, a faithful simulation of it) produces from a project profile.

- **The Jira project structure** at [PCO](https://lateapexllc.atlassian.net/jira/software/projects/PCO/boards/1): 8 build epics + 5 flagship stories + ~50 filler tickets + 13 subtasks; plus 6 future-work epics + 19 tasks + 63 subtasks tracking unfinished features (added 2026-04-27), distributed across 10 components. This is the output of running the orchestrator's blueprint workflow on the project's own profile (M4 in v6 §28). Where M4 isn't yet implemented end-to-end, the seeding script in [`scripts/demo/seed-jira.py`](../../scripts/demo/seed-jira.py) faithfully simulates the executor's output shape.
- **The Confluence space structure** at [ACO](https://lateapexllc.atlassian.net/wiki/spaces/ACO): 35 pages organized in 8 sections. Same provenance: blueprint output, faithfully simulated by [`scripts/demo/seed-confluence.py`](../../scripts/demo/seed-confluence.py) where the executor (M6b) isn't yet live.
- **The cross-linking between Jira and Confluence.** Every epic links to its v6 spec section; every flagship ticket links to its driving ADR; every Confluence centerpiece page lists its backing Jira tickets. The cross-link discipline is itself a generated output of the orchestrator's design — not me hand-stitching links.

The dogfooding frame is the demonstration. If this approach were broken — if the orchestrator produced outputs incompatible with how engineers actually work — you'd see it in the Jira project. The fact that the project looks coherent is the validation.

## Where the boundaries blur

There are places where the categories overlap honestly:

- **Code review.** Claude Code reviews code it has written. I review it back. The arbiter is the v6 spec.
- **ADR drafting.** I make the decisions. Claude Code drafts the prose. I edit.
- **Test writing.** I specify test cases. Claude Code implements them. I review for coverage gaps.
- **Documentation.** Most prose in [`docs/`](../) was drafted by Claude Code from my outlines. The spec (v6) and the partner-integration synthesis was a collaboration where I steered structure and Claude Code filled detail.
- **This very page.** I outlined what to say and the categorical structure (Designed / Orchestrated / Generated). Claude Code drafted the prose. I edited for tone and accuracy. That's a fair description of how a substantial fraction of the docs were produced.

I'm comfortable with these blurred lines because:

1. The decisions are mine.
2. The outputs are reviewed.
3. A reader who wanted to test this could pick any architectural decision and ask me to defend it — the answer would match what's documented because I made the call.
4. The boundary is documented in the open. If I were trying to misrepresent the work, hiding this page would be the move.

## What I'd be uncomfortable with

If someone built this without a v6-style spec — generating tickets and pages from a prompt and shipping them without architectural decisions — that would be a different artifact and I wouldn't represent it as engineering work.

The spec is the artifact. The implementation is the proof the spec is buildable. The generated output is the proof the implementation works. Each layer earns its trust from the layer above.

## A test you can run

If you suspect a particular decision was just regenerated from a prompt without judgment, ask me to defend it. Specifically:

- "Why ed25519 instead of secp256k1 for the audit signing?"
- "Why ADF as default with a flag, instead of choosing one and committing?"
- "Why pglite instead of just running Postgres in dev?"
- "Why the genesis-block special case instead of seeding with a synthetic prior?"
- "Why is the multi-tenant runway in v6 §7.3 deliberately unimplemented?"

Each of these has a one-paragraph answer that maps to specific tradeoffs I weighed. If the answer feels rehearsed, that's because I rehearsed it — I expect interviewers to ask, and I want to be ready.

## Closing

The risk in 2026 is that AI-assisted portfolios look indistinguishable from authored ones from the outside. The way to make them distinguishable is to be explicit about the boundaries and to engineer the work so the human judgment is visible at every level.

This page is part of that engineering. Every other page in the portfolio backs it up.
