Visualization plan — eighteen places where a picture beats a paragraph

The doc tree (docs/sdlc/) currently leans heavily on prose. This plan identifies 18 specific spots across the suite where a static SVG figure would carry concepts the prose either belabors or fails to make precise. Each card below is a finished example, not a sketch — copy the SVG into the named markdown file and ship it. Tier 1 figures are load-bearing; without them readers reconstruct the diagram in their head and get it wrong. Tier 2 figures resolve a frequent reviewer question. Tier 3 figures are nice-to-have for skim readability.

Editorial principles

  • Each figure must replace prose, not annotate it.
  • If a reviewer asks the same question twice in a doc, that's a figure.
  • Static SVG only — no interactivity, no JS dependencies. Versioned with the docs.
  • One typeface (IBM Plex), one ink color, semantic palette consistent across all 18.

Production notes

  • Each card links to a finished example. Right-click → "View source" to lift the SVG.
  • Drop into the markdown file as <figure><svg>…</svg><figcaption>…</figcaption></figure> using the caption from each example page.
  • Figures are referenced from the prose by ID (V1, V2, …) so cross-doc links survive renumbering.

What's intentionally missing

  • No homepage hero / marketing visuals — this is internal SDLC.
  • No screenshots of UI mocks — atl-mcp's UI is the MCP protocol surface; tools API docs cover it.
  • No animated diagrams. Documentation lives in PRs and PDFs; movement doesn't survive the trip.
  • No infra-as-code topology — that lives in the deploy repo, not the spec.
